# LAIA Path Resolver (Atlas)

Atlas is the live path resolver of the LAIA ecosystem â€” a single registry in
`~/.laia/config.yaml` defines short aliases for every project directory, and
a daemon (`laia-pathd`) keeps the aliasâ†’path mapping fresh by watching the
filesystem. When a directory is renamed or moved, every consumer (bash
scripts, systemd units, Python code, external programs via the symlink atlas)
keeps working without manual updates.

The name fits the role: an atlas is the map of where each thing lives. The
symlink farm at `~/.laia/atlas/` is the literal embodiment â€” one symlink per
alias, always pointing to the current location.

## At a glance

| Concept | Where it lives | What it does |
|---|---|---|
| **Registry** | `~/.laia/config.yaml` (`paths:` block) | Source of truth â€” aliases with `${paths.X}` references |
| **Daemon** | `laia-pathd` | Resolves aliases, watches filesystem, regenerates snapshots |
| **Snapshot** | `~/.laia/.env.paths` | `bash`-sourceable `export LAIA_X=â€¦` lines (regenerated on every change) |
| **Socket** | `~/.laia/pathd.sock` | Unix domain socket, JSON-RPC, used by Python clients |
| **Atlas (symlinks)** | `~/.laia/atlas/<alias>` | One symlink per alias, always pointing to current target |
| **State** | `~/.laia/state/path-cache.json` | Daemon state with inode tracking and change history |
| **CLI** | `laia-path` | Talk to the daemon: `resolve`, `list`, `env`, `status`, `doctor`, `reload` |

## Defining a new path

1. Edit the `paths:` block of `~/.laia/config.yaml`:
   ```yaml
   paths:
     # ... existing entries ...
     myservice: ${paths.laia_root}/services/myservice
   ```
2. The daemon detects the config change within ~2 seconds (or run
   `laia-path reload` to force it) and:
   - Regenerates `~/.laia/.env.paths` (adds `export LAIA_MYSERVICE="â€¦"`).
   - Creates `~/.laia/atlas/myservice` symlink.
   - Persists the new alias to `~/.laia/state/path-cache.json`.
3. Consumers (bash, Python, systemd) pick it up automatically the next time
   they look it up.

## Consuming paths

### From bash

```bash
source ~/.laia/.env.paths
echo "$LAIA_AGORA"
cd "$LAIA_AGORA"
```

The snapshot is regenerated by the daemon on every change. Long-running
shells can re-source on demand:

```bash
laia-path env > /tmp/env.paths && source /tmp/env.paths
```

### From systemd (units)

```ini
[Unit]
# When a registered alias moves, the daemon writes a pending-restart marker
# (see phase 5). The operator applies them with: laia-path apply-restarts
X-LaiaPathDeps=agora

[Service]
EnvironmentFile=-/home/laia-hermes/.laia/.env.paths
WorkingDirectory=/home/laia-hermes/LAIA/services/agora-backend
ExecStart=${LAIA_AGORA_VENV}/bin/uvicorn app.main:app --host 127.0.0.1 --port 8088
```

Notes:
- `EnvironmentFile=-â€¦` (with the leading `-`) makes systemd tolerate the file
  not existing yet (degraded mode).
- `WorkingDirectory` cannot expand env vars â€” keep it literal. The daemon
  detects when it diverges from the alias and queues a restart marker.

### From Python

```python
from laia_paths import get_path, all_paths

agora: Path = get_path("agora")
everything: dict[str, str] = all_paths()
```

`get_path` follows a tiered fallback:
1. Env var `LAIA_<KEY>` if set (instant).
2. Parse `~/.laia/.env.paths` (still no daemon round-trip).
3. Resolve `~/.laia/config.yaml` directly (last resort, daemon down).

### From external programs (editors, scripts)

Use the symlink farm. Each alias has a symlink at `~/.laia/atlas/<alias>`:

```bash
cd ~/.laia/atlas/agora     # always lands in the current agora dir
```

The daemon keeps these symlinks synchronized.

## CLI reference (`laia-path`)

| Command | What it does |
|---|---|
| `laia-path resolve <key>` | Print the resolved path for one alias. |
| `laia-path list` | Print all aliasâ†’path pairs (tab-separated). `--json` for JSON. |
| `laia-path env` | Print `export LAIA_X="â€¦"` lines (for `source <(laia-path env)`). |
| `laia-path status` | Daemon uptime, watch count, config mtime. |
| `laia-path doctor` | Validate every registered path exists. Exit 1 if any missing. |
| `laia-path reload` | Force re-read of `config.yaml` (also triggered by `SIGHUP`). |

When the daemon is down, `laia-path` falls back to resolving `config.yaml`
directly â€” same answers, no event push.

## Installing the daemon

```bash
# 1. Copy the user-mode systemd unit
mkdir -p ~/.config/systemd/user
cp /home/laia-hermes/LAIA/infra/systemd/laia-pathd.service ~/.config/systemd/user/

# 2. Enable and start
systemctl --user daemon-reload
systemctl --user enable --now laia-pathd.service

# 3. Verify
laia-path status
laia-path doctor
```

## What happens when a directory moves

Suppose someone renames `services/agora-backend/` â†’ `services/agora-api/`:

1. The kernel emits an inotify `MOVED_FROM` + `MOVED_TO` on the parent dir.
2. `laia-pathd` (subscribed via `watchdog`) wakes up, finds the alias whose
   `current_path` matched the old target, and updates it.
3. `~/.laia/.env.paths` is regenerated atomically (write to `.tmp`, rename).
4. `~/.laia/atlas/agora` symlink is recreated pointing to the new path.
5. `~/.laia/state/path-cache.json` records the transition (with timestamp
   and reason `fs-rename`).
6. If any systemd unit had `X-LaiaPathDeps=agora`, a pending-restart marker
   is written to `~/.laia/state/pending-restarts.json` (phase 5).
7. The operator runs `laia-path apply-restarts` when ready.

No `sed -i`, no manual edits, no broken services. The registry is the source
of truth; everything downstream catches up automatically.

## Troubleshooting

**Daemon won't start.** Check `journalctl --user -u laia-pathd.service`.
Common causes: `~/.laia/config.yaml` has a circular reference (daemon logs
`CircularPathReference`); socket path not writable.

**A path shows as `missing` in `doctor`.** The directory has been deleted
and the daemon couldn't auto-detect a new location (auto-discovery rules
are a future addition). Either:
- Recreate the directory, then `laia-path reload`.
- Edit `~/.laia/config.yaml` to point to the new location.

**A consumer still uses an old path.** Either:
- A bash session sourced `.env.paths` before the change â†’ re-source.
- A long-running process cached the path â†’ restart it (manual or via
  `apply-restarts` if it has `X-LaiaPathDeps`).
- The consumer hardcoded the path â†’ grep for it and switch to the alias.

## Implementation details

- **Resolver** (`laia_paths.py`): pure function, no side effects, importable
  from anywhere on PYTHONPATH (declared in `.laia-core/pyproject.toml`'s
  `py-modules`).
- **Daemon** (`infra/pathd/`): asyncio + watchdog. Polling fallback at 2s
  intervals for `config.yaml`; inotify on parent dirs of registered paths.
- **IPC**: line-delimited JSON-RPC over Unix socket (mode `0600`).
- **State**: JSON file written atomically (write-temp + rename).
- **Atomicity guarantees**: `.env.paths` is written via the same atomic
  pattern, so `source ~/.laia/.env.paths` never reads a half-written file.

## Adding a new alias to an existing service

Concrete example: you want to add `${paths.workspace_store}` because a new
component needs the path.

1. Add to `~/.laia/config.yaml`:
   ```yaml
   paths:
     workspace_store: ${paths.laia_root}/workspace-store
   ```
2. The daemon picks it up within ~2s; verify:
   ```bash
   laia-path resolve workspace_store
   echo $LAIA_WORKSPACE_STORE   # works after `source ~/.laia/.env.paths`
   ```
3. In bash code:
   ```bash
   source ~/.laia/.env.paths
   ls "$LAIA_WORKSPACE_STORE"
   ```
4. In Python:
   ```python
   from laia_paths import get_path
   p = get_path("workspace_store")
   ```
